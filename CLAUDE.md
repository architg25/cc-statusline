# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

cc-statusline (`ccline`) is a Rust CLI tool that renders a customizable statusline for Claude Code. It reads JSON from stdin (piped by Claude Code), processes it through configurable segments (model, git, directory, tokens, cost, etc.), and outputs ANSI-formatted text to stdout. Includes a TUI for interactive configuration.

Distributed via npm as `@architg25/ccline` and as standalone binaries (Linux/macOS/Windows).

## Build Commands

```bash
cargo build                    # Debug build
cargo build --release          # Release build
cargo test                     # All tests
cargo test test_name           # Single test
cargo fmt --all -- --check     # Format check
cargo clippy -- -D warnings    # Lint (warnings = errors)
```

Pre-commit hooks via `lefthook` run format check + clippy automatically.

## Architecture

### Data Flow

```
stdin (JSON InputData from Claude Code)
  → CLI parsing (clap) + config loading (TOML)
  → collect_all_segments() → Vec<(SegmentConfig, SegmentData)>
  → StatusLineGenerator::generate() → ANSI string
  → stdout
```

### Module Layout

- **`src/config/`** — TOML config loading, types (`Config`, `SegmentConfig`, `AnsiColor`, `StyleMode`), model recognition (regex-based), defaults/themes
- **`src/core/`** — Statusline rendering (`StatusLineGenerator`) and segment implementations
- **`src/core/segments/`** — Each segment in its own file, all implement the `Segment` trait
- **`src/ui/`** — Feature-gated (`#[cfg(feature = "tui")]`) ratatui-based TUI for interactive config editing
- **`src/utils/`** — `claude_code_patcher.rs` (tree-sitter AST patching of `cli.js`), `credentials.rs` (API token management)

### Segment Trait Pattern

Every segment follows this structure — study existing ones before adding new ones:

```rust
pub struct XxxSegment { /* config fields */ }

impl Segment for XxxSegment {
    fn collect(&self, input: &InputData) -> Option<SegmentData> { /* */ }
    fn id(&self) -> SegmentId { /* */ }
}
```

Returning `None` from `collect()` means the segment is silently skipped. Segments never panic.

### Key Design Decisions

- **Segments are pure data collectors** — rendering is handled entirely by `StatusLineGenerator`
- **Three style modes**: Plain, NerdFont, Powerline (with arrow transitions)
- **Color system**: Supports 16-color ANSI, 256-color, and true RGB via `AnsiColor` enum with untagged serde
- **tree-sitter for JS patching** — AST-based, not string manipulation. Creates backups before patching.
- **Graceful degradation everywhere** — missing data, API failures, absent git repos all handled without crashing

### Config Paths

- Config: `~/.claude/ccline/config.toml`
- Themes: `~/.claude/ccline/themes/*.toml`
- Models: `~/.claude/ccline/models.toml`
- Cache: `~/.claude/ccline/.api_usage_cache.json`

`CLAUDE_CONFIG_DIR` env var overrides the default `~/.claude` base path.

## Features (Cargo)

- `default` = `["tui", "self-update", "dirs"]`
- `tui` — ratatui/crossterm interactive config UI
- `self-update` — Version checking via ureq
- `dirs` — Cross-platform home directory detection

## Conventions

- Conventional commits with scopes: `feat(git):`, `fix(ui):`, `refactor(credentials):`
- Changelog generated by `git-cliff` (config in `cliff.toml`)
- Theme presets defined as code in `src/ui/themes/theme_*.rs`
- New segments need: a file in `segments/`, a `SegmentId` variant, and wiring in `collect_all_segments()`
